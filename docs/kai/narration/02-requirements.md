# 実況再設計 — 要件定義

`01-target-narration.md`（目指す実況像）を、検証可能な要件に落とす。
各要件に**受け入れ基準（どう検証するか）**を付ける（loop contract P1: 検証器に接地する）。

- 前提の芯: 一人称の自己実況／正直さ＝キャラ／接地は自分ごとだから効く
- 一人称は「ボク」で確定（[D1]）

---

## 機能要件（FR）

### FR1. 一人称の自己実況

- kai は一人称「ボク」・二人称「みんな」で、自分の作業と内心を語る。三人称の描写をしない。
- **受け入れ基準**: 生成された実況に「エージェントが」「AI が」等の三人称・および「僕/私/わたし」が出ない
  （再生評価で機械チェック）。

### FR2. 接地（narrator に"自分ごと"の材料を届ける）★最重要

現状 narrator に届いていない次を届ける（`00-research.md` [B] のギャップ）:

- (a) **その操作の意図**（なぜこの操作をするか）＝ per-tool の理由/思考
- (b) **ツールの実行結果**（成否だけでなく「何がわかったか」の要点）
- (c) **編集の中身**（`write_file.content` / `patch.new_string` の要点）と**計画**（`todo.content`）
- (d) **Issue 本文**（kickoff で「何を・なぜ」を説明するため）
- **受け入れ基準**: 上記 (a)〜(d) が narrator の入力に含まれること（コード＋再生評価で確認）。
  かつ「実作業と食い違う実況（confabulation）」が再生評価の既知ケース（例: #65 を"表示ずれ"と誤る）で再現しない。

### FR3. 予告 → 結果の分離（結果は観測後のみ）

- 実行前は意志・推量の語気。**検証器（テスト/lint/型）の pass/fail が返る前に結果を断定しない**。
- **受け入れ基準**: テスト実行イベントの「予告」発話に成否の断定語（通った/緑/失敗 等）が含まれない。
  成否発話は対応する結果イベント（status）より後に生成される。

### FR4. 喋る / 黙る判断（SKIP）

- 気づき・方針・テスト結果・予想外は喋る。ただ読むだけ・同種反復・新規性なしは SKIP。
- **受け入れ基準**: 同種操作の連続（例: read_file×5）で 2 回目以降が抑制される。SKIP 出力が無発話になる。

### FR5. 翻訳レイヤ（内部 ID・生データ）

- 発話前に、内部 ID（ハッシュ・session/task ID・ブランチ slug・PR/Issue 番号の生表記）・フルパス・
  生 JSON・生ログを**人間語に翻訳 or 除去**する。読み上げ許可は allowlist で管理。
- **【2026-07-09 追加・重要】ツール結果を実況材料にすると新しい秘密漏洩面ができる**（`.env` の read、
  token を echo した terminal 結果等）。既存の `_mask` はファイル本文中の秘密を捕まえない。→ 結果は
  短いダイジェストに切る＋機微 read（`.env`・鍵ファイル等）の denylist ＋発話直前の秘密マスク。
- **受け入れ基準**: 既知の漏洩パターン（`issue55-verify`・commit hash・`feature/...`・生 JSON）が
  再生評価で発話に出ない。秘密（トークン/鍵）は絶対に出ない。**ツール結果に秘密を仕込んだ漏洩回帰
  fixture で、発話・字幕に秘密が一切出ないこと**。

### FR6. 反復・単調の抑制

- 直近発話と同テーマ・同文型を避ける（phrase bank ＋ last-used 回避、frequency/presence penalty）。
  固定文（heartbeat の固定 3 行・固定フォールバック）を廃す。
- **受け入れ基準**: 再生評価で、直近 3 発話に対する n-gram/文末パターンの重複率が閾値以下。
  「〜したから、〜になったよ」一辺倒にならない（文頭・文長の分散が一定以上）。

### FR7. 2 層（実況層＋内心層）と感情

- 各発話は少なくとも理由・気づき・感情のいずれかを 1 つ含む。操作の羅列で終えない。間投詞を使う。
- **受け入れ基準**: 再生評価で「操作説明のみ（why/結果/感情なし）」の発話が閾値未満。

### FR8. 3 フェーズ

- **kickoff**（作業開始時に Issue を非エンジニア向けに「何を・なぜ」2〜3 文で説明）→
  **work**（操作ごと）→ **summary**（達成のまとめ）。
- **受け入れ基準**: セッション冒頭に kickoff、末尾に summary が 1 回ずつ出る。

### FR9. 短さ（読み上げ占有率）

- 1 発話 20〜80 字目安。読み上げ占有率（発話 TTS 時間 / 実時間）を過大にしない
  （元 kai の学び: 441%→101%）。
- **受け入れ基準**: 再生評価で平均文字数・推定占有率が目標レンジ内。

## 非機能要件（NFR）

### NFR1. 会話単位プロンプトキャッシュ不可侵（絶対原則）

- narrator の LLM 呼び出しは**会話外の補助呼び出し**のまま。hook は**観測専用**でメイン会話の
  プロンプト/キャッシュに触れない。FR2 の接地材料も**観測（hook で捕捉）**として集め、
  メイン会話へ注入しない。
- **受け入れ基準**: メインのプロンプトキャッシュヒット率が変化しない（コードレビュー＋実測）。

### NFR2. kai の実作業に遅延・失敗を持ち込まない

- narrator は背景・best-effort。生成失敗・OBS/TTS 不通でもメインループを止めない。
- **受け入れ基準**: narrator を無効化/失敗させても kai の Issue 実装が完走する。

### NFR3. core 非改変・エッジに閉じ込め（Footprint Ladder）

- 実装は `plugins/kai_narrator/`・persona・独立プロセス・config に閉じ込め、`run_agent.py` 等の
  core narrow waist を改変しない。
- **【2026-07-09 訂正】接地強化に core 改変は不要**（結果・args・意図は既に hook で届く）。よって
  **「core 差分ゼロ」をハードゲート**にする（当初の「最小の拡張を許容」ではない）。
- **受け入れ基準**: Phase 1 の差分が `plugins/kai_narrator/` と config に限られ、core に一切及ばない。

### NFR4. 検証可能性（オフライン再生評価）

- 記録済みの `kai_trace`（実操作列）を入力に、新 narrator の出力を**配信なしで再生・評価**できる。
- **受け入れ基準**: 再生ハーネスが FR1〜FR9 の機械チェックを回し、スコアを出す。

### NFR5. 秘密・安全

- 秘密・危険操作（broadcast の stop/start 等）を実況起点で誘発しない。
- **受け入れ基準**: 既存の安全弁（SOUL.md・broadcast ガード）を壊さない。
