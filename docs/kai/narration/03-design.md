# 実況再設計 — 基本設計

`02-requirements.md` を満たす設計。**背骨は「改善ループ（オフライン評価環境）」**。
実況の良し悪しを配信せずにスコアで測り、改善を回せるようにする（loop contract P1）。

> **【2026-07-09 改訂】** この文書の一部は懐疑レビュー（Opus × Codex）で訂正され、
> `04-implementation-plan.md` が正典。主な訂正: (1) **接地に core 改変は不要**（結果・args・意図は
> 既に hook で届く）＝ §2.1 の「core 改変を要するなら…」は不要、(2) 冒頭 Issue 説明（kickoff）は
> **本体ではなく観測者**が担う（本体はターン終端で 1 回しか喋らない）、(3) 観測者は**接地事実の翻訳＋SKIP**に
> 絞り意図・感情を創作させない、(4) ツール結果を材料にする**秘密漏洩対策**が必須、(5) eval は temp=0。
> 実装は §3 の「ハーネス先」ではなく **Phase 1（プラグインだけの接地修正）を先**にする。

## 0. 背骨 — 改善ループ（先に検証環境を作る）

```text
 ①本番相当の操作ログ            ②実況生成             ③評価
 ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
 │ narration     │   │ narrator      │   │ evaluator     │
 │ fixture       │──▶│ (candidate)   │──▶│ 機械チェック   │──▶ スコア
 │ (op列+接地)   │   │  = 生成ロジック │   │ + LLM ジャッジ │      │
 └──────────────┘   └──────────────┘   └──────────────┘      │
        ▲                                                     │
        └──────────── プロンプト/ロジックを調整して再実行 ◀──────┘
```

- **配信不要**。記録済みの操作ログ（fixture）を食わせ、実況を生成し、`01-target-narration.md` の
  理想カタログ・要件（FR1〜9）に対してスコアリングする。
- 実況の変更（プロンプト・翻訳・反復抑制・接地）はすべて**このハーネスで回帰確認**してから実機へ。
- **だからハーネスを最初に作る**（§3 実装順序）。これが無いと「良くなった」を検証器に接地できない。

## 1. アーキテクチャ決定 — 案 C（ハイブリッド）

`00-research.md` [B] の A/B/C から **C** を採る。理由: confabulation・無音・キャッシュ不可侵を同時に解ける。

| 経路                                    | 担当                                       | 実装                                                        | キャッシュ                    |
| --------------------------------------- | ------------------------------------------ | ----------------------------------------------------------- | ----------------------------- |
| **本体の声**（source=agent_response）   | kickoff の Issue 説明・重要な判断・summary | SOUL.md 強化で kai が一人称で意図を語る（既に発話済み経路） | 出力なので不可侵を守る        |
| **観測者**（source=narrator, 接地強化） | 操作ごとの実況・結果反応・沈黙埋め         | `plugins/kai_narrator/` を接地強化＋一人称化                | 会話外の補助 LLM のまま不可侵 |

要は「**なぜ・要点は本体が自分の声で**、**沈黙埋め・結果反応・逐次実況は観測者が接地された状態で**」。

## 2. コンポーネント設計

### 2.1 接地パイプライン（hook 観測で材料を集める。キャッシュ不可侵）

現状の欠落（Issue 本文・意図・ツール結果・編集中身）を、**メイン会話に触れず hook 観測で**集める。
元 kai の `AgentActivity` バッファ＋`formatActivityBuffer()`（thought＋tool_use＋tool_result を
一人称ログ化）を移植する。

ターン単位で「意図（そのターンの assistant テキスト/reasoning）」と「その操作」を結合する:

```text
1 ターン = [assistant の思考/テキスト(=意図)] + [tool_call…] → [tool_result…]
  pre_tool_call  : ツール開始（実行中マーク）
  post_tool_call : ツール名 + args(content/new_string/todos 含む) + result要点(≤Nchar) + status  ← 結果を追加捕捉
  post_llm_call  : そのターンの意図テキスト + conversation_history（既に渡っている・未活用）
        ▼ これらを turn 単位で AgentActivity に束ね、一人称ログへ整形
  「ボクが今考えていること: …／今やったこと: …／今つかんだこと(結果): …」
```

- **接地に必要な最小拡張**（NFR3 に従い個別評価）:
  - `_ARG_KEYS` に `content`/`old_string`/`new_string`/`todos` を追加（編集の中身・計画を拾う）。
  - `post_tool_call` の payload に**ツール結果の要点ダイジェスト**（境界長・秘密マスク済み）を追加。
    → core への影響が出る場合は、plugin 側で結果を観測する方法を優先し、最小性をレビュー承認。
  - Issue 本文は kickoff 時に `gh issue view`／ブラウザ操作の文脈から取得（既に kai が open している）。
- **キャッシュ不可侵**: 上記はすべて**観測**。メイン会話のメッセージ列・system プロンプトには一切書かない。

### 2.2 narration fixture（本番相当の操作ログ＋接地）とレコーダー

改善ループの入力。フォーマット（JSONL・1 行 1 イベント、接地込み）:

```jsonc
{
  "phase": "kickoff|work|summary",
  "ts": "…",
  "issue": { "number": 65, "title": "…", "body": "…" }, // kickoff のみ
  "turn_intent": "preflight に後片付け項目を足す。口伝を書き残したい", // そのターンの意図
  "tool": "patch",
  "args": { "path": "…", "new_string": "…" }, // 中身込み
  "result": { "status": "ok", "digest": "3行追記。verify 未実行" },
}
```

- **取得方法（2 系統）**:
  1. **レコーダー**: ドライラン中に §2.1 の接地パイプラインが集めた材料を、そのまま fixture として保存する
     （＝本番と同じ操作・同じ接地。最も忠実）。
  2. **ブートストラップ**: 既存の実リハーサル trace（2026-07-06 / 07-08、24〜37 操作）＋当時のリポジトリ状態から
     代表 fixture を数本手組みし、既知の失敗ケース（#65 を"表示ずれ"と誤る）を**回帰テストとして固定**。
- fixture は `docs/kai/narration/fixtures/` か kai_trace 配下に版管理し、増やしていく。

### 2.3 narrator 生成（3 フェーズ・一人称ログ・プロンプト）

- **3 フェーズ**（元 kai 準拠）: kickoff（Issue を非エンジニア向けに 2〜3 文）→ work（操作ごと）→ summary。
- 入力は §2.1 の一人称ログ（意図＋操作＋結果）。プロンプトは `01-target-narration.md` の
  人格・境界・3 原則（予告/結果分離・2 層・SKIP）を反映。
- **反捏造**: `<log>`/`<body>` 等 XML タグでデータと指示を分離し「payload にない原因を推測しない／
  材料が無ければ SKIP」を明示。結果の断定は result イベント到達後のみ（FR3）。
- LLM は補助タスク `narration` のまま。`frequency_penalty 0.6 / presence_penalty 0.3` を付ける（反復抑制）。

### 2.4 発話後処理（翻訳・sanitize・反復抑制）

生成文を発話（speechd）へ渡す前に通す固定パイプライン（元 kai の実運用対策を移植）:

- **翻訳レイヤ**: 内部 ID（hash・session/task ID・`feature/...` slug・`#65` の生表記）→ 人間語 or 除去。
  読み上げ allowlist。フルパス→basename、URL/生 JSON→除去。
- **sanitize / fallback**: `sanitizeSpeech`（path/絵文字/バッククォート除去）、`safeSpeechFallback`
  （JSON 断片の字幕流出を検出し差し替え）、秘密マスク（既存＋強化）。
- **反復抑制**: 直近発話（deque）に対する文末・n-gram 重複を弾く（phrase bank ＋ last-used 回避）。
  固定 heartbeat 3 行・固定フォールバック文を**廃止**し、接地された生成 or SKIP に置換。

### 2.5 評価器（machine checks ＋ LLM ジャッジ）

fixture で生成した実況を、要件に対して採点する。

- **機械チェック**（正規表現・ヒューリスティック）:
  - FR1 三人称/禁止一人称の非出現、FR3 予告に成否断定が無い、FR5 ID/秘密の非漏洩、
    FR6 直近との重複率、FR7 「操作のみ」発話率、FR9 平均文字数・占有率。
- **LLM ジャッジ**（隔離コンテキスト）: 各発話を「操作・結果」と突き合わせ、
  ①接地しているか（confabulation でないか）②一人称の自己実況か ③理由/感情があるか ④理想カタログに近いか
  を採点。**#65 の"表示ずれ"のような食い違いを検出**できることを回帰で担保。
- 出力: FR ごとの pass/率 ＋ 総合スコア ＋ 悪かった発話の抜粋（改善の手がかり）。

## 3. 実装順序（★検証環境を先に）

1. **ハーネス骨組み（先）**: fixture フォーマット確定 → 既存 trace から代表 fixture を数本手組み
   （#65 失敗ケース含む）→ 現 narrator をそのまま食わせて**現状スコアのベースライン**を出す。
   （＝改善の基準線。ここまでで「悪さ」を数値化）
2. **接地パイプライン**: §2.1（AgentActivity ＋ `_ARG_KEYS` 拡張＋結果ダイジェスト）。レコーダーで
   忠実 fixture を録れるようにする。
3. **生成・後処理の刷新**: §2.3・2.4（3 フェーズ・一人称ログ・翻訳・反復抑制・penalty）。
4. **ループを回す**: ハーネスでスコアが基準線を上回るまでプロンプト/ロジックを調整（オフライン）。
5. **実機ドライラン**: [5] 検証。実配信相当のドライランでスクショ＋trace＋発話を残し、人＋懐疑レビュー。
6. **対応 / Issue 化**: 上記を粒度で Issue に割る。

## 4. リスクと検証

- **忠実性リスク**: fixture が本番とズレると評価が当てにならない → レコーダーで実ドライランから録るのを基本にし、
  手組み fixture は回帰用の少数に留める。
- **接地拡張が core に及ぶリスク**（NFR3）: ツール結果の観測が core 改変を要するなら、plugin 側観測を優先し、
  最小差分を懐疑レビュー（deep-reasoner / codex）で承認してから。
- **LLM ジャッジの信頼性**: ジャッジ自身の誤判定 → 機械チェックと併用し、既知ケースの回帰で較正。
- **P4**: この設計と接地拡張は重要変更。実装前に隔離コンテキストの懐疑レビューを通す。
