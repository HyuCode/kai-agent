# kai ドキュメントガイド

このディレクトリは、AITuber「kai」の要件・設計・運用・検証記録を管理する正典です。
読者が上から順に辿れるよう、ディレクトリとファイル名を番号で統一しています。

## 推奨する読み順

| 順番 | カテゴリ                                          | 内容                                     |
| ---- | ------------------------------------------------- | ---------------------------------------- |
| 1    | [`01-product/`](01-product/00-index.md)           | プロダクト要件、MVP、ロードマップ        |
| 2    | [`02-architecture/`](02-architecture/00-index.md) | システム全体と共通コンポーネントの設計   |
| 3    | [`03-features/`](03-features/00-index.md)         | アバター、実況、自動配信などの機能別仕様 |
| 4    | [`04-operations/`](04-operations/00-index.md)     | 配信、プリフライト、VMデプロイの運用手順 |
| 5    | [`05-quality/`](05-quality/00-index.md)           | Loop Engineering、検証、品質保証         |
| 6    | [`06-evidence/`](06-evidence/00-index.md)         | リハーサルや実機検証の記録               |
| 7    | [`99-templates/`](99-templates/00-index.md)       | 新しい設計書を作るためのテンプレート     |

## 正典の優先順位

文書間で内容が競合した場合は、次の順で判断します。

1. `01-product/01-requirements.md` の全体要件
2. `02-architecture/01-system.md` のシステム共通設計
3. `03-features/` の機能別要件・設計
4. `04-operations/` の現行運用手順
5. `06-evidence/` の検証時点における記録

新しい決定が上位文書と衝突する場合は、下位文書だけで上書きせず、上位の正典も同じ変更で更新します。
検証記録は事実の証跡ですが、要件や設計を定義する文書ではありません。

## 番号規則

- `00-*`: その階層の索引・読み方
- `01-*`: 要件または最初に読む本文
- `02-*` 以降: 設計、計画、運用手順を読む順に配置
- `90-*`: fixtureなど本文を補助するデータ
- `99-*`: テンプレート

番号は表示順のためのものです。要件ID（`F-xx`、`Gx`、`FR-Lx`）は変更せず、文書を移動してもトレーサビリティを維持します。

## 更新ルール

- 新規機能は、要件を `03-features/<番号>-<機能名>/01-requirements.md` に置き、必要なら `02-design.md` を追加します。
- 全機能に共通する決定は `02-architecture/01-system.md` に置きます。
- 手順やコマンドは `04-operations/` に置き、要件・設計へ埋め込みません。
- 実機の結果やリハーサル所感は `06-evidence/` に置き、正典と混ぜません。
- 各文書の先頭にステータス、前提、関連する要件IDを記載します。

> この再編は文書の配置とナビゲーションを整理するものです。既存文書間の意味上の矛盾は、上記の優先順位に従って別途解消します。
