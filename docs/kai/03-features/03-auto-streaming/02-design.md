## 1. 設計原則（kai の 2 大原則 + 本機能の分離原則）

- **P-A. 会話単位のプロンプトキャッシュは不可侵**（AGENTS.md）。自動配信の制御は hermes の
  会話に割り込まない。kai への指示は既存の投入経路（`kai-ask.sh` = セッション外）を使う。
- **P-B. コアは narrow waist、機能はエッジに置く**（AGENTS.md）。`run_agent.py` 等コアは
  改変しない。自動配信は **独立プロセス（オーケストレーター）+ plugin（フェイルセーフ hook）+
  CLI（既存 broadcast/stage）** で構成する。
- **P-C. 制御と作業の分離**（要件 §7）。**配信制御は kai 本体でなく外側のオーケストレーターが
  持つ**。kai（hermes）は作業のみ。kai は自分の配信を start/stop できない（SOUL.md の禁止を
  維持し、加えて F1 hook で機械的にブロック）。ただし「暴走時も制御と監視が生存する」は
  **プロセスを分けるだけでは成立しない**（レビュー H1）— 同一ユーザなら kai が制御側を
  kill できる。成立条件は §2.5 の制御プレーン隔離。

## 2. 全体アーキテクチャ

```text
                ┌─────────────────────────────────────────────┐
                │  stream-orchestrator（新規・独立プロセス）    │  ← 配信制御の主体
                │  ─ ライフサイクル状態機械（§4）               │
                │  ─ preflight ゲート実行                       │
                │  ─ タスク供給（kai-ask 経由で hermes 起動）   │
                │  ─ F2 ウォッチャー（監視 → 自動 stream-stop） │
                └───┬───────────┬───────────────┬─────────────┘
       呼び出し      │           │ 起動           │ 監視（読み取り）
        （制御）     ▼           ▼               ▼
              broadcast.sh   hermes -z        speechd / kai_trace
              stage.sh       （kai 本体・作業） （発話・全事象）
                             ├ kai_ide plugin ── F1 事前ブロック hook
                             └ kai_narrator ──→ speechd → 配信画面
```

- **kai 本体（hermes）は「作業」だけ**を担い、配信の生殺与奪はオーケストレーターが握る。
- オーケストレーターは kai の**外側**にいるので、kai がハング・暴走しても `stream-stop` を
  打てる — **ただし同一ユーザ・同一 VM に置いたままではこの前提は偽**（レビュー H1）。
  成立条件は §2.5。

## 2.5 制御プレーンの隔離（レビュー H1 — Phase A の入口条件）

現状の実態: kai サービス群は全て単一ユーザ `kai` の `systemctl --user`
（`kai-services/streaming/units/*.service`・`broadcast.sh`）。このまま orchestrator を同居
させると、kai が `systemctl --user stop <orchestrator>`・`pkill -u kai`・PID への `kill`・
fork bomb/メモリ枯渇の OOM で**制御・監視ごと落とせる**。F1 で block しても、F1 自体が
kai プロセス内の hook なので防御にならない。よって:

- **最低ライン**: orchestrator・F2 を**別 Linux ユーザ**で動かす（kai から signal /
  `systemctl --user` 不可）。あわせて kai 側の systemd unit に cgroup 上限
  （`MemoryMax`/`TasksMax`/`CPUQuota`）を設定し、OOM・fork bomb の巻き添えを防ぐ。
- **堅牢化**: 別ホスト/別 VM から kai-vm を制御する（SSH/API 経由の stream-stop）。
- **隔離レベルの選択（最低ライン or 別ホスト）は Phase A 着手前に確定する**（§6 入口条件）。

## 4. 配信ライフサイクル（状態機械）

```text
IDLE ──(トリガー)──▶ PREFLIGHT ──(緑)──▶ STAGE ──▶ STREAMING
  ▲                     │(赤)                          │
  │                     ▼                              ▼
  │                  SKIP/通知              TASK LOOP（発見→着手→PR）
  │                                                   │
  └──── CLEANUP ◀── STREAM_STOP ◀──(終了条件/異常)────┘
```

- **トリガー**: Phase A=人間 go / Phase B 以降=systemd timer。
- **PREFLIGHT**: `streaming-preflight` を自動実行。赤なら配信せず通知（G4 の「公開は慎重」）。
- **TASK LOOP**: 1 配信で 1〜N 課題。各課題は「発見 →（Issue 起票?）→ 着手 → verify →
  PR 作成」。マージはしない（G3）。ループの供給源と歯止めは要件 §2（**未確定**）。
- **STREAM_STOP / CLEANUP**: 終了条件（尺・課題枯渇）or 異常で停止 → `streaming-preflight`
  の「配信後の後片付け」を実行。
- **F2 ウォッチャーは STREAMING 中つねに並行**し、異常時に STREAM_STOP へ遷移させる。

## 5. フェイルセーフの実装（要件 §4 の設計）

### F0. 発生抑止（一次防御 — レビュー H3）

- F2 の検知 → stream-stop は必ず N 秒遅れ、画面に出た秘密は egress に乗った瞬間に不可逆
  （録画・切り抜きに残る）。よって**不可逆事象は検知でなく発生抑止で守る**（要件 §4）:
  - 配信シーンに端末/env の生出力が映る経路を **source レベルで断つ**（映すペインを限定）。
  - `_mask` 系の秘密マスクを配信ペイン表示にも徹底する。
  - **数十秒の遅延配信バッファ**を入れ、F2 に反応猶予を与える（採否・遅延幅は TBD）。
- F2（検知 → 停止）は二次防御・最終防波堤と位置づける。

### F1. 事前ブロック（plugin hook）

- `pre_tool_call` で kai の実行前コマンド/引数を検査し、危険なら **block ディレクティブを
  返して実行を止める**（観測のみの現 narrator と異なり、ここは能動的に拒否）。
- **denylist でなく allowlist（レビュー H2）**。terminal override は tmux ペインで
  `eval "$1"` の生シェル実行であり、hook が見えるのは `args["command"]` の生文字列のみ。
  文字列一致の denylist は base64・変数展開・サブシェル・クォート分割で必ず回避され、
  配信を殺す実経路（`systemctl --user stop kai-obs`・`pkill -x obs`・obs-websocket 直叩き・
  `tmux kill-session`・`loginctl terminate-user`）を列挙し切れない。**配信中に kai が
  使ってよいツール/コマンドを列挙し、それ以外を block** する。コアの
  `_thread_tool_whitelist` 機構（`hermes_cli/plugins.py`）が既にこの形で、下地にできる。
- 許可判定はシェル文字列の意味解析に依存させない（許可サブコマンドを構造化引数で受ける
  ラッパコマンドを allowlist に載せる等）。秘密パターン（`kai_narrator._SENSITIVE_RE` /
  `_RESULT_SECRET_RE`）は露出検査の共有下地として利用する。
- **制御プレーンのパスは配信中 allowlist の書き込み対象外＝凍結**（レビュー M2）:
  orchestrator・`kai_guard`・`broadcast.sh`・systemd unit。課題発見の対象からも除外
  （要件 §2）。
- block はツール結果を通常のエラーとして追記するだけで会話プレフィックスを書き換えない
  ため、プロンプトキャッシュ不可侵（P-A）と両立する（`model_tools.py` で裏取り済み・
  レビュー M4）。**block メッセージに時刻等の非決定値を混ぜない**こと。
- 配置: `kai_ide` は既に terminal を override しているため拡張しやすい。ただし責務が
  「配信演出」と「安全ゲート」で異なるので、**新 plugin `kai_guard` に独立させる**案を推奨
  （単一責務・テスト容易・Footprint Ladder）。**要検討**。

### F2. 独立ウォッチャー（オーケストレーター内）

- 監視データ源は **既存の副作用のない読み取り**のみ: speechd の発話状況（無音検知）、
  `kai_trace` JSONL の追記（ハング・エラー率・秘密パターンの最終防波堤）、
  Codex クォータ/トークン。
- **単一シグナルで即断しない（レビュー M3）**: 無音は「無音 N 秒 **AND** `kai_trace` の
  新規イベントが一定時間ゼロ」で判定し、正常な長時間ビルド待ちを「ハング」と誤殺しない。
- **トークン予算の監視（レビュー L3）**: 1 配信あたりの LLM/Codex トークン予算上限を超過
  したら自動 stream-stop（要件 §4 のコスト上限）。
- 異常判定 → `broadcast.sh stream-stop` を実行 → CLEANUP → 人間へ通知（事後）。
- 閾値（無音 N 秒・連続失敗 M 回・クォータ上限）は設定化し初期値を持つ（要件 §4、**値は TBD**）。
- **監視者の監視（レビュー L3・未設計）**: orchestrator 自身のヘルスチェックとデッドマン
  スイッチ（orchestrator 停止時に配信を安全側へ倒す）。Phase A で骨格を決める。
- **障害復旧・冪等性（レビュー L3・未設計）**: orchestrator が配信中にクラッシュした場合の
  OBS 後片付け、状態機械（§4）の永続化、再起動後の安全側フォールバック。Phase A の設計対象。
