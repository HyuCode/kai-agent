## 1. ゴール（確定 — 2026-07-09 オーナー合意）

- **G1. 完全無人配信。** 最終的に、人間の開始トリガーなしに配信を開始・運転・終了する。
  段階的に自律度を上げる（半自動 → スケジュール無人 → 完全無人）前提で、**ゴールは完全無人**。
- **G2. kai が課題自体を発見する。** 何をするかを人間が与えるのでなく、kai が自分で次の
  改善課題を見つけて着手する（自己改善エージェント）。
- **G3. main へのマージは人間承認を維持（loop contract P3 を死守）。** 無人化しても、
  main への反映ゲートだけは人間が握る。kai は PR 作成・CI 緑まで。**これが安全の芯** —
  kai が暴走・誤判断しても main（＝次回以降の自分自身のコード）は保護される。
- **G4. 外部への公開・不可逆アクションは人間承認（G3 の一般化）。** main マージ（G3）は
  この原則の一例。**読み取り・分析・下書き作成までは kai 自律、外部に届く/取り消しにくい
  操作（SNS 投稿・動画公開・チャンネル設定変更）は人間が最終 GO** を出す。炎上・スパム・
  秘密の外部流出（#77 の延長）を人間ゲートで止める。

## 1.5 kai の活動領域（方向確定・詳細 TBD — 2026-07-09 オーナー合意）

kai の自律範囲を、コード改善だけでなく**自分のメディア運営**まで広げる。自己改善ループが
コンテンツにも回る（伸びた配信を分析 → 次の企画へ）。2 系統:

### 開発（自己改善）

コード・テスト・ドキュメント。成果は PR（main は人間マージ = G3）。詳細は §2。

### メディア運営

G4 の「読み取り自律 / 公開は人間承認」で線引きする:

| タスク         | 性質                   | 承認         | 必要実装・メモ                                                                                                                   |
| -------------- | ---------------------- | ------------ | -------------------------------------------------------------------------------------------------------------------------------- |
| チャンネル分析 | 読み取り               | 自律         | YouTube Data/Analytics API。視聴数・維持率・コメント傾向を分析し次の企画に反映。**一番手前で価値**                               |
| 動画投稿       | 生成は自律／公開は承認 | 公開のみ人間 | アーカイブから切り抜き（trace の見どころで切り出し点選定）→ タイトル/説明生成（LLM 得意）→ アップロードは人間 GO。編集の質は課題 |
| SNS 発信       | 文面は自律／投稿は承認 | 投稿のみ人間 | 「今日の成果」を X 等へ。文面生成は自律、投稿は人間 GO。認証トークンは #77 の秘密管理対象                                        |
| チャット応答   | 双方向                 | —            | §C（視聴者インタラクション）。プロンプトインジェクション対策必須（#74 の XML エスケープと同種）                                  |

**TBD**: どの API から着手するか（分析が最有力）。認証トークン（YouTube/X OAuth）の

（PR と同様に「下書きを人間が確認して GO」）。

### G1〜G3 から導かれる制約

- 完全無人＝**誰も見ていない**。異常の検知と停止を人間に頼れない → フェイルセーフ（§4）が
  「あれば良い」でなく**生命線**。preflight（開始前・静的）だけでなく**配信中の動的監視**が要る。
- 人間マージ × 無人連続作業 → **マージ待ち PR が積み上がる**。次タスクのベースが古くなる／
  依存する変更が未マージで詰む問題の扱いを決める（§3 TBD）。
- kai が課題発見 → 発見の源と**無限ループ防止の歯止め**が要る（§2 TBD）。

## 2. タスク供給 — kai の課題発見（方針確定・仕組みは TBD）

**確定**: kai がオープン Issue の消化でなく、**課題自体を発見**する（G2）。

**TBD（一緒に詰める）**:

- 発見の源は何か（候補、複数併用可）:
  - trace（`kai_trace`）の振り返り＝過去配信の失敗・非効率から改善点
  - verify / テストの穴（カバレッジ不足・TODO・FIXME・skip されたテスト）
  - upstream 追従で生じた不整合
  - 既存のオープン Issue（人間が種を蒔く余地は残すか？）
- 発見した課題を**一旦 Issue として起票してから着手**するか（証跡・人間の事前介入余地）、
  その場で着手するか。
- **歯止め**: 1 配信あたりの課題数上限／同じ領域を触り続けない／「やることが無い」ときの挙動
  （沈黙・待機・配信しない）。

**確定（レビュー M2）: 自己制御系は課題発見・改変の対象外。** 課題発見の源が自分の
narrator・オーケストレーター・フェイルセーフ・`broadcast.sh` を指すと、G3 で main を守っても
**稼働中の作業ツリー書き換え／サービス再起動経由で自分の制御系を壊せる**。配信中は制御プレーン
のパスへの書き込みを F1 で凍結し（設計 §5）、課題発見の対象からも除外する。

## 3. マージ待ち PR の積み上げ（方針確定・運用は TBD）

**確定**: マージは人間（G3）。kai は PR を作って次へ。

**確定（レビュー M1 — G1×G3 の構造矛盾はここで解く。数値のみ TBD）**:

未マージ PR が溜まる → 毎回古い main ベースで作業 → 同じ改善を再発見して重複 PR、という
飽和が G1 の実現可能性そのものに関わるため、TBD でなく運用規約として確定する:

- **未マージ PR の上限**: N 本溜まったら新規着手を停止して人間へ通知（N の値は TBD、
  初期値 3 を仮置き）。人間マージ律速の安全弁。
- **既マージ main ベースで小さく切る**: 未マージ PR に依存する後続課題は選ばず、独立した
  課題を main ベースで着手する（PR をスタックしない）。
- **発見済み課題の重複排除**: 発見した課題は Issue（または trace）に記録し、未マージでも
  「発見済み」として再発見・重複 PR を防ぐ。

## 4. フェイルセーフ（最重要・これから一緒に設計）

完全無人の生命線。**何を監視し・どの閾値で・どう止めるか**を詰める。

**確定（2026-07-09 レビュー反映）**:

- **制御プレーンの実隔離（レビュー H1）**: オーケストレーター・監視が kai と同一ユーザ・
  同一 VM にいる限り「外側だから常に止められる」は**成立しない**（kai が
  `systemctl --user stop`・`pkill -u kai`・メモリ枯渇の OOM で制御・監視ごと落とせる。
  現状の kai サービス群は全て単一ユーザ `kai` の `systemctl --user`）。**最低ラインは
  別 Linux ユーザ（kai から signal / `systemctl --user` 不可）＋ kai 側の cgroup 上限
  （`MemoryMax`/`TasksMax`/`CPUQuota`）。堅牢化は別ホスト/別 VM**。隔離レベルの決定は
  Phase A の入口条件（設計 §6）。
- **不可逆事象は検知でなく発生抑止で守る（レビュー H3）**: 秘密が画面に出た瞬間フレームは
  配信 egress に乗って不可逆（録画・切り抜きに残る）。検知 → stream-stop は必ず N 秒遅れる
  ため**二次防御**と位置づけ、一次防御は「そもそも映さない」— 配信シーンに端末/env が映る
  経路を source レベルで断つ・`_mask` の配信表示への徹底・遅延配信バッファで反応猶予を確保
  （採否 TBD）。
- **コスト上限（レビュー L3）**: 1 配信あたりの LLM/Codex トークン予算上限を設け、超過で
  自動 stream-stop（値は TBD）。無人ゆえの暴走課金の歯止め。

**TBD の骨子**:

- 監視対象（候補）: 秘密の画面露出（上記のとおり二次防御・最終防波堤）／無音継続
  （**無音単独では判定しない** — 正常な長時間ビルド待ちを誤殺しないため trace 追記の停滞と
  AND 条件にする。レビュー M3）／hermes のハング・暴走／verify 赤ループ（直せず同じ所に固執）／
  Codex クォータ・コスト暴走／不適切・オフブランド発言。
- 停止の権限: 完全無人なので**自動 stream-stop が既定**。人間通知は事後。
- 停止の粒度: 配信だけ止める／作業も止める／セッションを畳む。
- preflight（静的・開始前）と動的監視（配信中）の役割分担。
- **監視者の監視（レビュー L3）**: 監視側自身が死んだことを誰が検知するか。オーケストレーター
  のヘルスチェック・デッドマンスイッチ。
- **障害復旧・冪等性（レビュー L3）**: オーケストレーターが配信中にクラッシュした場合、誰が
  OBS を後片付けするか。状態の永続化と再起動後の安全側フォールバック。
